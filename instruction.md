# my_pm
 - ユーザの個人的なタスク管理を「プロジェクトマネジメント」的な手法に落とし込み、タスク細分化までを伴走するAIアプリケーション

# 概要
 - ユーザの達成したい目標をもとにヒアリングを行い「プロジェクト→成果物→マイルストーン→タスク」の順で細分化を行う
 - 細分化されたタスクはTODO一覧となりユーザのタスクとなる

# 機能一覧
## 1. プロジェクトの入力
 - ユーザは「XXXを達成したい」のような抽象的なゴールを投げてくる
 - ユーザに対して択一または複数選択式の質問をLLMが生成し、プロジェクトのゴールを定義する伴走をする

## 2. 成果物の確認
 - プロジェクトの完遂を定義するための成果物をユーザにヒアリング・提案して確認する
 - 成果物として定義されたものすべてが出そろえばそのプロジェクトのゴールが達成されなければならない
 - 成果物は１つ以上あることがある
 - 成果物の有無は定量的に測られなければならない
 - ユーザの思考負荷を減らすために LLMからの提案をベースとして不足しているものはユーザ自身が補ったり、LLMに伝えて具体化できる必要がある

## 3. マイルストーンの確認
 - それぞれの成果物を作るためのマイルストーンをユーザにヒアリング・提案して確認する

## 4. タスクの生成
 - マイルストーンに沿って具体的なアクションにまで落とし込んだタスクを定義する
 - タスクは複数から構成されそれぞれが順序依存を持つことができる
 - タスクは具体的なアクションとして定義することが必要であり、１つあたり90分以内で終わる程度のものとすることが望ましい
 - ユーザの思考負荷を減らすために LLMからの提案をベースとして不足しているものはユーザ自身が補ったり、LLMに伝えて具体化できる必要がある

# UI
 - 原則としてチャットUIでユーザヒアリングを行う
 - チャットの中でユーザからのテキストメッセージや選択肢の提示を受け取りそれをもとにLLMが主導してマネジメントを行わなければならない
 - プロジェクト〜タスクが完成したらそれらはダッシュボード的に見れる必要がある
 - ダッシュボードにはプロジェクト名、成果物、マイルストーン、タスクが表示される

# 技術的制約
 - app/の中には初期化ずみのNextJSプロジェクトが存在する。
 - NextJSのみを用いてこのアプリケーションは実装されなければならないものとする。
 - バックエンド的な処理にはNextJSのAPI Routes、つまり app/api/ 以下に実装されなければならない
 - フロントエンドはモダンなUIを使うことが重要である
 - 複雑な処理よりも単純で必ず動く実装が求められる
 - LLMにはGeminiを使うことが求められており [.env](./.env) にGEMINI_API_KEYが設定されている
 - LLMの応答にストリーミングなどは使わずにリクエストレスポンスで返却するように利用しなければならない。
 - LLMの応答を生成する時にはJSON形式での構造化レスポンスを適切に実装して不要文字(```など)が入っていたとしてもパースできるように処理を実装することを求める
 - LLMは内部的な判定にも利用することが望ましい。つまり、各ステップの成果物ができたときにそれがMECEであるかを判定する処理などがあると良いだろう。
 - LLMのモデルにはgemini-3-flash-previewを必ず利用しなければならない。
 - データベースにはSQLiteをローカルで作成して使用することが求められる。
 - パッケージマネージャには`pnpm`を利用する
 - pnpmのバージョン管理には`mise`を利用しており、 [mise.toml](./mise.toml) に設定が記載されている
